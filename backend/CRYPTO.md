## Главная идея: zero-knowledge

Сервер **не может** расшифровать данные пользователя. Он хранит зашифрованные блобы и не знает ключ.
Ключ шифрования живёт только в памяти браузера пользователя и никогда не покидает его устройство.

Аналогия: сервер — это банковская ячейка. Он не знает, что внутри. Ключ — только у клиента.

## Алгоритм 1: PBKDF2 — получение ключа из пароля

**Где:** `frontend/src/crypto/kdf.ts`
**Кто вызывает:** браузер пользователя при входе и регистрации

### Что это такое

Пользователь вводит мастер-пароль. Это строка — ненадёжный материал для шифрования.
PBKDF2 превращает пароль в надёжный криптографический ключ.

Принцип простой: берёт пароль + случайную соль, и **очень много раз** прогоняет через SHA-256.
Много раз — это намеренно, чтобы атакующий не мог быстро перебрать пароли: каждая попытка занимает время.

### Как используется в проекте

Из одного мастер-пароля PBKDF2 создаёт **два разных вывода** — специально, чтобы они никогда не совпали:

| Что получаем | Итераций | Зачем |
|---|---|---|
| **Ключ шифрования** | 600 000 | Шифровать/расшифровывать данные в хранилище |
| **Хеш для проверки** | 100 000 | Отправить на сервер, чтобы подтвердить правильность пароля |

Ключ шифрования — никогда не покидает браузер.
Хеш для проверки — отправляется на сервер вместо пароля.

### Параметры
- Хеш-функция: SHA-256
- Соль: 32 случайных байта, генерируются при регистрации
- Соль хранится на сервере в открытом виде (это нормально — соль не секрет)

## Алгоритм 2: AES-256-GCM — шифрование данных

**Где:** `frontend/src/crypto/aes.ts`
**Кто вызывает:** браузер пользователя при сохранении и чтении записей

### Что это такое

AES-256-GCM — симметричное шифрование. Один и тот же ключ используется и для шифрования, и для расшифровки.

- **AES** (Advanced Encryption Standard) — сам алгоритм блочного шифрования. Стандарт де-факто, одобрен правительством США для секретных данных.
- **256** — длина ключа в битах (32 байта). Чем длиннее — тем надёжнее.
- **GCM** (Galois/Counter Mode) — режим работы. Важная добавка: GCM не только **шифрует**, но и **подписывает** данные (auth tag). Если кто-то изменил зашифрованный блоб — расшифровка упадёт с ошибкой. Это защита от подделки.

### Что хранится в базе данных

Когда пользователь сохраняет запись (логин + пароль + URL), браузер шифрует всё это и отправляет три поля:

| Поле в БД | Что это | Размер |
|---|---|---|
| `encryptedData` | Зашифрованный JSON с данными | Переменный |
| `iv` | Инициализирующий вектор (случайный, новый при каждом шифровании) | 12 байт |
| `authTag` | Тег аутентификации (подтверждает целостность) | 16 байт |

**IV** — это как одноразовый номер. При каждом шифровании генерируется заново.
Если бы IV не менялся, одинаковые данные давали бы одинаковый шифртекст — это плохо.

Сервер хранит всё это, но **не может расшифровать** — ключа у него нет.

## Алгоритм 3: Argon2id — хеширование на сервере

**Где:** `backend/config/packages/security.yaml`, `backend/src/Service/Auth/AuthService.php`
**Кто вызывает:** сервер при регистрации и входе

### Что это такое

Argon2id — победитель соревнования Password Hashing Competition 2015. Современная замена bcrypt.

Принцип: берёт строку (пароль или его производный хеш), применяет к ней тяжёлую функцию и отдаёт хеш.
«Тяжёлую» — в смысле затратной по памяти и времени. Намеренно, чтобы перебор был медленным.

### Что именно хешируется

Пользователь никогда не отправляет мастер-пароль напрямую. Вместо него браузер отправляет
`masterPasswordHash` — вывод PBKDF2 (см. алгоритм 1).

Сервер хеширует этот `masterPasswordHash` через Argon2id и сохраняет в поле `passwordHash`.
При следующем входе — сравнивает присланный хеш с сохранённым.

### Параметры в проекте
```yaml
# config/packages/security.yaml
algorithm: argon2id
memory_cost: 65536   # 64 MB оперативной памяти на каждый расчёт
time_cost: 4         # 4 прохода
```

Такие параметры означают: один расчёт хеша требует 64 МБ RAM и занимает ~0.3–1 секунду.
Для пользователя — незаметно. Для атакующего с миллионом попыток — непосильно.

## Алгоритм 4: SHA-256 — хеширование refresh-токена

**Где:** `backend/src/Service/Auth/TokenService.php`
**Кто вызывает:** сервер при создании и проверке refresh-токенов

### Что это такое

SHA-256 — быстрая хеш-функция. Одностороннее преобразование: из хеша нельзя получить исходные данные.

Здесь SHA-256 используется **не для паролей** (для паролей — только Argon2id, потому что SHA-256 слишком быстрый).
Для токенов это нормально: токен — длинная случайная строка (64 символа), перебрать невозможно.

### Зачем это нужно

Refresh-токен — длинная случайная строка, выдаётся пользователю при входе.
Если хранить его в БД в открытом виде, то при утечке базы атакующий получит активные сессии.

Поэтому в БД хранится только SHA-256 от токена. Пользователю выдаётся оригинал.
При проверке: SHA-256(присланный токен) == stored_hash.

## Алгоритм 5: RSA — подпись JWT-токенов

**Где:** `backend/config/packages/lexik_jwt_authentication.yaml`, ключи в `backend/config/jwt/`
**Кто вызывает:** сервер при создании и проверке access-токенов

### Что это такое

RSA — асимметричное шифрование. Два ключа: приватный (секретный) и публичный.

- Приватный ключ: подписывает токен при выдаче
- Публичный ключ: проверяет подпись при запросах

### Зачем это нужно для JWT

JWT-токен — это JSON с данными пользователя (email, время истечения и т.д.), подписанный RSA.
Сервер при каждом запросе проверяет подпись — если токен не подделан, подпись сойдётся.

Параметры:
```yaml
# config/packages/lexik_jwt_authentication.yaml
secret_key: '%kernel.project_dir%/config/jwt/private.pem'
public_key: '%kernel.project_dir%/config/jwt/public.pem'
token_ttl: 3600  # JWT живёт 1 час (фактически access-токен 15 мин — из кода AuthController)
```

## Итого: что происходит при входе

```
1. Пользователь вводит email + мастер-пароль в браузере

2. Браузер запрашивает у сервера: GET /api/v1/auth/kdf-params?email=...
   Сервер отдаёт: соль + параметры PBKDF2 (итерации, алгоритм)

3. Браузер вычисляет локально:
   - encryptionKey    = PBKDF2(пароль, соль, 600 000 итераций) ← НИКОГДА не покидает браузер
   - masterPasswordHash = PBKDF2(пароль, соль, 100 000 итераций) ← отправляется на сервер

4. Браузер отправляет: POST /api/v1/auth/login { email, masterPasswordHash }

5. Сервер проверяет: Argon2id.verify(masterPasswordHash, stored_passwordHash)

6. Если совпало — сервер выдаёт:
   - access_token (JWT, подписан RSA, живёт 15 минут)
   - refresh_token (случайная строка, в БД хранится её SHA-256 хеш, живёт 30 дней)

7. Браузер хранит:
   - accessToken → в памяти (не в localStorage!)
   - refreshToken → в localStorage
   - encryptionKey → в памяти (не в localStorage!)
```

## Итого: что происходит при сохранении записи

```
1. Пользователь вводит: логин + пароль + URL

2. Браузер шифрует: AES-256-GCM(JSON.stringify(данные), encryptionKey)
   Получает: { encryptedData, iv, authTag }

3. Браузер отправляет зашифрованный блоб на сервер

4. Сервер сохраняет в БД encryptedData + iv + authTag
   Сервер не знает ключа → не может расшифровать

5. При чтении — браузер получает блоб и расшифровывает локально
```
