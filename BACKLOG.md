# PassFort — Бэклог бэкенда

Список функционала для добавления в бэкенд, упорядочен по приоритету.

---

## Критически важно (безопасность)

### 1. Смена мастер-пароля
Нет ни эндпоинта, ни логики. При смене мастер-пароля нужно обновить `salt`, `kdfParams`, `masterPasswordHash` — клиент перешифровывает все `encryptedData` с новым ключом. Без этого пользователь не может обезопасить аккаунт при компрометации пароля.

### 2. Двухфакторная аутентификация (TOTP)
Не реализована. TOTP (Google Authenticator / Aegis) — стандарт для менеджеров паролей.
Нужны: генерация secret, QR-код, эндпоинт верификации кода, backup-коды, флаг `is2faEnabled` в `User`.

### 3. Блокировка аккаунта при брутфорсе
Rate limiting есть (5 попыток / 15 мин), но блокировки аккаунта нет. После N неудачных попыток — временная блокировка (`User.isActive = false`) + email-уведомление + запись в `SecurityLog`.

### 4. Управление активными сессиями
`RefreshToken` хранит IP и User-Agent, но эндпоинтов для управления нет.
- `GET /api/v1/user/sessions` — список активных сессий
- `DELETE /api/v1/user/sessions/{id}` — отозвать конкретную сессию

## Высокая приоритетность (функциональность)

### 5. Email-верификация при регистрации
`EmailNotificationMessage` определён, но помечен TODO и не реализован. Без верификации email любой может занять чужой адрес. Нужен: токен верификации в БД, эндпоинт подтверждения, ресенд письма.

### 6. Email-уведомления о событиях безопасности
Kafka-транспорт `kafka_notifications` есть, handler-а нет. Уведомлять пользователя о: входе с нового IP/устройства, смене пароля, удалении аккаунта, новой сессии.

### 7. API просмотра Security Log для пользователя
`SecurityLog` пишется в БД, но пользователь не может его просмотреть.
- `GET /api/v1/user/security-log` — история событий с пагинацией

### 8. Смена email с верификацией
`UserService::updateEmail()` меняет email мгновенно без подтверждения. Нужен flow: запрос смены → письмо на новый адрес → подтверждение по токену.

## Средняя приоритетность (UX и функциональность)

### 9. Поиск и фильтрация записей
`VaultItemRepository::findByVault()` возвращает все записи без фильтров. Нужны query-параметры:
- `?type=login` — по типу записи
- `?category=uuid` — по категории
- `?q=github` — поиск по `titleHint`
- `?favorite=true` — только избранные

### 10. Пагинация для vault items
При большом количестве записей возвращается весь массив. Нужна пагинация (`page`, `limit`) или cursor-based.

### 11. Экспорт/импорт хранилища
- `GET /api/v1/vaults/{id}/export` — экспорт зашифрованного JSON (бэкап)
- `POST /api/v1/vaults/{id}/import` — импорт (совместимость с Bitwarden/KeePass форматами)

### 12. Bulk-операции с записями
- `DELETE /api/v1/vaults/{id}/items` с body `{"ids": [...]}` — массовое удаление
- `PATCH /api/v1/vaults/{id}/items/move` — перенос записей между категориями

## Низкая приоритетность (инфраструктура)

### 13. Cron-очистка истёкших refresh-токенов
`RefreshTokenRepository::deleteExpiredTokens()` реализован, но нигде не вызывается. Нужна Symfony Console Command + cron или Messenger ScheduledMessage.

### 14. Rate limiting на vault-эндпоинтах
Сейчас ограничен только `/auth/*`. Нужен лимит на запись/изменение items для защиты от злоупотреблений API.

### 15. HIBP-прокси
`GET /api/v1/security/hibp-range/{prefix}` — проксирование запроса к `api.pwnedpasswords.com`.
Актуально если нужно скрыть факт обращения к HIBP от сетевого наблюдателя. Сама проверка утечки пароля — только на фронтенде (zero-knowledge архитектура не позволяет делать это на сервере).

### 16. Health-check эндпоинт
`GET /api/v1/health` — статус БД, Redis, Kafka. Нужен для мониторинга и docker healthcheck.
